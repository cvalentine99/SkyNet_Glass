# Skynet Glass — Comprehensive Bug Fix & Polish Prompt for Claude Code

> **Context:** Skynet Glass is a React + Express + tRPC + MySQL dashboard for monitoring the Skynet firewall on ASUS Merlin routers. It connects to the router via SSH, fetches `stats.js` (generated by Skynet's `Generate_Stats()`), parses the JavaScript variables, and displays them in a glass-cockpit dashboard. The app runs on a local Ubuntu server on the same LAN as the router. Authentication is disabled (LAN-only mode). The codebase is at `~/skynet-glass` (or `~/SkyNet_Glass`).

## CRITICAL: Database Connection Issue (Fix First)

The `.env` file has `DATABASE_URL=mysql://skynet:YOUR_DB_PASSWORD@127.0.0.1:3306/skynet_glass` with a **placeholder password**. Every single tRPC query that touches the database fails with:

```
Failed query: select ... from `skynet_config` limit ? params: 1
```

**This is the root cause of ALL "missing data" issues across every page.** The ipsets dropdown is empty, the logs page shows nothing, the dashboard KPIs work only because they come from SSH (stats.js), not the DB.

### Fix:
The `install.sh` script (382 lines, already in the repo) handles this automatically — it generates a random password, creates the MySQL user, writes `.env`, runs migrations, and sets up systemd. But if the user already has MySQL set up, the script should detect the existing `skynet` user and reset its password. **Review `install.sh` and make sure it handles the "MySQL already installed, skynet user already exists" case gracefully** — it should `ALTER USER` instead of `CREATE USER` if the user exists.

After fixing install.sh, also add a **startup health check** in `server/_core/index.ts` (or wherever the Express server starts) that tests the DB connection on boot and logs a clear error message like:

```
[FATAL] Cannot connect to MySQL. Check DATABASE_URL in .env. Error: <actual error>
```

instead of silently failing on every query.

## Bug 1: Ipsets Page — Dropdown Does Not Populate

**File:** `client/src/pages/Ipsets.tsx`

**Symptom:** The IP Address dropdown on `/ipsets` shows nothing. The user sees "No banned entries found" even though the router has 39,326 IPs banned.

**Root Cause:** The `getBlacklist` tRPC procedure calls `fetchIpsetData()` which calls `getSkynetConfig()` which queries the `skynet_config` table — and that fails because of the DB connection issue above. So `result.error` is set and the procedure returns `{ entries: [], summary: null, error: "..." }`.

**Secondary Issue:** Even when the DB works, `fetchIpsetData()` runs:
```bash
cat /opt/share/skynet/skynet.ipset 2>/dev/null || { ipset save Skynet-Blacklist; ipset save Skynet-BlockedRanges; ipset save Skynet-Whitelist; ipset save Skynet-WhitelistDomains; } 2>/dev/null
```

The `skynet.ipset` file on ASUS Merlin can be **very large** (39K+ entries × ~100 bytes = ~4MB+). The SSH command has a 30-second timeout which may not be enough. Also, the entire file is transferred in one shot and parsed client-side.

**Fix:**
1. Fix the DB connection (see above) — this is the primary fix.
2. In `server/skynet-fetcher.ts`, increase the timeout for `fetchIpsetData` from 30000 to 60000ms.
3. Consider adding pagination at the SSH level: `grep -c '^add' /opt/share/skynet/skynet.ipset` first to get the count, then `head -n 2000` for the first page.
4. The Ipsets page already has client-side pagination (PAGE_SIZE = 100), which is good.

## Bug 2: Logs Page — Shows "No log entries found"

**File:** `client/src/pages/Logs.tsx`

**Symptom:** `/logs` shows 0 entries, "No log entries found. Connect your router in Settings to see live syslog entries."

**Root Cause:** Same as Bug 1 — `fetchSyslog()` calls `getSkynetConfig()` which fails on the DB query. The syslog data comes from:
```bash
tail -n ${maxLines} /tmp/syslog.log 2>/dev/null | grep -i "PRIOR\|skynet\|DROP\|REJECT"
```

**Fix:**
1. Fix the DB connection (primary fix).
2. The syslog grep pattern `"PRIOR\|skynet\|DROP\|REJECT"` should also include `"BLOCKED"` and `"DENY"` which some Skynet versions use.
3. In `server/skynet-fetcher.ts` line ~480, the syslog command should be:
```bash
tail -n ${maxLines} /tmp/syslog.log 2>/dev/null | grep -iE "PRIOR|skynet|DROP|REJECT|BLOCKED|DENY"
```

## Bug 3: Dashboard — Blocked Connections Chart Shows 0

**File:** `client/src/components/charts/BlockedConnectionsChart.tsx`

**Symptom:** The Blocked Connections bar chart shows "0" for both Inbound and Outbound, even though the KPI cards show 39,326 IPs banned.

**Root Cause:** The `inboundBlocks` and `outboundBlocks` KPI values come from `SetHits1` and `SetHits2` in `stats.js`. These are the **block hit counters** (how many packets were blocked), NOT the ban count. If Skynet hasn't blocked any packets since the last stats generation (or if the log was recently cleared), these will be 0. This is **correct behavior** — the chart is showing real data.

**Fix:** The chart already has an empty state message (added recently). Verify it says something like: "Skynet does not yet have blocking data. Try 'Update Stats' on the router." This is informational, not a bug.

## Bug 4: Dashboard — Block Trends Chart is Empty

**File:** `client/src/components/charts/TrendChart.tsx`

**Symptom:** The Block Trends section is completely empty — no data points.

**Root Cause:** The trend chart reads from the `skynet_stats_history` table, which stores one row per successful stats fetch. Since the DB connection is broken, no history has been saved. Even after fixing the DB, it will take multiple polling cycles before enough data points exist to draw a trend line.

**Fix:**
1. Fix the DB connection (primary fix).
2. After the DB works, the `fetchStatsFromRouter()` function in `skynet-fetcher.ts` should be saving snapshots to `skynet_stats_history` on each successful fetch. Verify this is happening.
3. The TrendChart should show a helpful message when there are fewer than 2 data points: "Collecting data... Trends will appear after multiple polling cycles."

## Bug 11: Syslog Parser — Missing Patterns

**File:** `server/skynet-syslog-parser.ts`

**Symptom:** Some Skynet log entries may not be parsed because the regex patterns don't cover all Skynet log formats.

**Context:** Skynet writes to syslog with patterns like:
```
Feb 28 12:34:56 router-hostname kernel: PRIOR1IN=eth0 OUT= MAC=... SRC=1.2.3.4 DST=192.168.50.1 ... PROTO=TCP SPT=12345 DPT=443
```

The key fields are: `SRC`, `DST`, `PROTO`, `SPT`, `DPT`, `IN=`, `OUT=`, and the `PRIOR` prefix indicates Skynet blocked it.

**Fix:** Review `server/skynet-syslog-parser.ts` and ensure the regex handles:
1. `PRIOR1IN` (inbound block, priority 1)
2. `PRIOR1OUT` (outbound block)
3. `PRIOR2IN` / `PRIOR2OUT` (priority 2, IoT blocks)
4. Both `PROTO=TCP` and `PROTO=UDP` and `PROTO=ICMP`
5. Lines where `SPT` or `DPT` may be missing (ICMP has no ports)

## Architecture Notes for Context

### How Data Flows

```
Router (ASUS Merlin + Skynet)
  │
  ├── /ext/skynet/stats.js     ← Generated by Skynet's Generate_Stats()
  │     Contains: SetBLCount1(), SetHits1(), DataInPortHits.unshift(), etc.
  │     Fetched via: SSH cat /jffs/scripts/www_ext/skynet/stats.js
  │
  ├── /opt/share/skynet/skynet.ipset  ← The actual ban list
  │     Format: add Skynet-Blacklist 1.2.3.4 comment "BanMalware: abuse.ch"
  │     Fetched via: SSH cat /opt/share/skynet/skynet.ipset
  │
  ├── /tmp/syslog.log           ← Live firewall log
  │     Format: kernel: PRIOR1IN=eth0 ... SRC=x.x.x.x DST=y.y.y.y PROTO=TCP
  │     Fetched via: SSH tail -n 500 /tmp/syslog.log | grep ...
  │
  └── /jffs/scripts/firewall    ← Skynet CLI
        Commands: ban ip, unban, iot ban, iot unban, etc.
        Executed via: SSH /jffs/scripts/firewall <command>
```

### stats.js Variable Names (from Skynet source)

The `stats.js` file contains JavaScript functions and arrays. Here are the exact variable names:

**KPI Functions:**
- `SetBLCount1()` → IPs Banned (sets `blcount1` innerHTML)
- `SetBLCount2()` → Ranges Banned (sets `blcount2` innerHTML)
- `SetHits1()` → Inbound Blocks (sets `hits1` innerHTML)
- `SetHits2()` → Outbound Blocks (sets `hits2` innerHTML)
- `SetStatsDate()` → "Monitoring From <date> To <date>"
- `SetStatsSize()` → "Log Size - (2.8MB)"

**Chart Data Arrays (populated via `.unshift()`):**
- `DataInPortHits` / `LabelInPortHits` → Top 10 targeted inbound ports
- `DataSPortHits` / `LabelSPortHits` → Top 10 source ports
- `DataTIConnHits` / `LabelTIConnHits_IPs` / `LabelTIConnHits_Country` → Top 10 inbound blocks (by IP and country)
- `DataTOConnHits` / `LabelTOConnHits_IPs` / `LabelTOConnHits_Country` → Top 10 outbound blocks
- `DataTHConnHits` / `LabelTHConnHits_IPs` / `LabelTHConnHits_Country` → Top 10 HTTP(S) blocks
- `DataTCConnHits` / `LabelTCConnHits` → Top 10 blocked devices (LAN IPs)

**Connection Table Arrays:**
- `LabelInConn_IPs`, `LabelInConn_BanReason`, `LabelInConn_AlienVault`, `LabelInConn_Country`, `LabelInConn_AssDomains` → Last 10 inbound connections
- `LabelOutConn_*` → Last 10 outbound connections
- `LabelHTTPConn_*` → Last 10 HTTP connections

### Ipset File Format

Each line in `skynet.ipset`:
```
add Skynet-Blacklist 1.2.3.4 comment "BanMalware: abuse.ch"
add Skynet-BlockedRanges 5.6.7.0/24 comment "Country: CN"
add Skynet-Whitelist 8.8.8.8 comment "ManualWlistD: dns.google"
add Skynet-WhitelistDomains 142.250.0.0/16 timeout 604800 comment "CDN-Whitelist: Google"
```
